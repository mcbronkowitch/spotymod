# Bench run 2026-07-22 — `8668367`

Measured on a Daisy Seed (STM32H750). 480000000 Hz core clock, block size 96, dcache+icache, `-ffast-math -funroll-loops`. Block budget 960000 cycles.

## Verdict

**2x4 budget — go/no-go.** The full instrument at its worst case (8 voices, COLOR 4-note on both parts, all FX on, high diffusion, echo at max) costs **94 % of the block budget offline**, and **94 % measured inside a real audio callback**. The anchored figure is the one that decides. **Conclusion: the 2x4 architecture fits.** The anchored figure is under 100 % of the block budget.

**Cost per candidate, relative to one real spotymod voice.**

- `modal_voice` — 7.2x one real voice (44x a bare oscillator kernel)
- `string_voice` — 2.2x one real voice (14x a bare oscillator kernel)
- `resonator` — 6.6x one real voice (40x a bare oscillator kernel)
- `formant_osc` — 0.35x one real voice (2.1x a bare oscillator kernel)
- `vosim_osc` — 1.1x one real voice (6.7x a bare oscillator kernel)
- `harmonic_osc` — 0.57x one real voice (3.5x a bare oscillator kernel)
- `grainlet_osc` — 0.77x one real voice (4.7x a bare oscillator kernel)
- `z_osc` — 1.2x one real voice (7.3x a bare oscillator kernel)
- `variable_shape_osc` — 0.22x one real voice (1.4x a bare oscillator kernel)

**SRAM vs SDRAM.** The grain-read proxy (8 scattered interpolated stereo reads per sample, identical window in both regions) costs **5.3x** in SDRAM against SRAM. That is a bare access pattern, written before the sampler existed to stand in for it; the `sampler_win_*` pair below is the same contrast with the real engine around it. The Oliverb pair reads **1.3x**, and the shortened echo-style streaming walk **3.3x**.

**The texture deck.** Both parts on the sampler at its worst case — DENS 8, SIZE 0.05 (128-sample grains spawning every 16), MOTION 1 scattering reads over the whole 42 s record buffer, SCAN running, and instrument_worst's FX chain unchanged around it — costs **85 % of the block budget on the mean block and 107 % on the worst**, against **90 % / 94 %** for the same box with both parts on the synth. The mean is the cheaper of the two; the peak is not. **Conclusion: the texture deck fits on average and overruns on peaks.** The grain count itself is already capped (`kSpawnHeadroom`) and the ablations say the remainder is not a burst: with the MOTION and SIZE lanes quiet the cloud's per-block work is flat to 1 %, and the worst block is simply the ceiling's worth of grains plus the whole FX chain. Dropping either FLUX or the reverb from this patch brings it under 100 %; capping grains harder does not.

Solo, the same cloud is **17 %** mean / **20 %** peak, a musical setting (DENS 4, half-second grains, MOTION 0.5, playhead parked) is **10 %** mean, and running an overdub underneath the worst-case cloud costs **1.1x** its mean. SCAN driven at the VCV host's rate — every 16 samples, six times the engine's own control tick, the `std::pow` `SamplerEngine::set_scan`'s comment flags — reads **1.0x** against the same cloud without it. With the whole engine around it the record buffer's region costs **1.2x** (`sampler_win_sdram` over `sampler_win_sram`, identical settings and identical 8192-frame content), well under the bare proxy's figure above: the scheduler, window, filter and normalization are all region-blind and dilute it.

*Figures in this section are quoted to whole percentage points and two significant figures for ratios — honest to what this bench can actually resolve (intra-run jitter of roughly 1700 cycles on a 1.5M-cycle workload, and a cross-build layout shift that moved a 29K-cycle workload by about 7%). The tables below retain full measured precision.*

## What this run measures that the previous one did not

Three changes landed between `d230206` and this run, all of them removals of
work nothing reads. None of them touches an ear-tuned constant, a mapping curve
or a topology; the full desktop suite (541 cases) is green and the two headline
ratios below moved only because the denominators got cheaper.

| row | `d230206` avg / max | here avg / max |
|---|---|---|
| `instrument_worst` | 92.80 / 98.11 | 89.74 / 93.72 |
| `instrument_worst_taps` | 98.09 / **103.53** | 94.47 / **99.15** |
| `inst_sampler_worst` | 92.25 / **113.16** | 85.45 / **106.82** |
| `instrument_init` | 62.95 / 74.65 | 59.62 / 69.99 |
| `synth_4_voices` | 19.48 / 19.76 | 17.24 / 17.51 |
| `sampler_flow_worst` | 18.27 / 21.86 | 16.76 / 20.01 |
| anchored `instrument_worst` | 93.10 / 98.09 | 90.02 / 93.99 |

**`instrument_worst_taps` is now inside the budget** — it was the one synth row
over it. **`inst_sampler_worst` is still over**, at 107 % on the worst block; it
came down 6.3 points but the remaining overrun is what the previous run already
established it to be (steady FX-chain load, not a spawn burst), and no
measurement here changes that diagnosis.

Where the three changes came from, and how they were found: the diagnosis was
*counted*, not timed. A desktop harness linked the engine sources directly,
reproduced `setup_inst_worst` exactly, and counted call sites per block. That
turned up 192 libm `sinf`, 60 `powf`, 16 `expf` and 12 `log10f` per 96-sample
block on the two-part instrument — and, more usefully, 768 `Svf::Process` calls
each computing five outputs and a cubic drive term to use one output and a
provably-zero drive. See the three commits for the per-change numbers.

One first guess died on that harness before it cost a hardware run: PartFx's
`_grit_applied` change guard was suspected of missing on every sample under
modulation, which would have put two `expf` per sample per part on the audio
path. The counter says the guard misses **zero** times per block in the worst
case. It is not a leak.

**Checksums.** The `sampler_*` and instrument rows' checksums moved; the
`synth_*` rows' did not. That is expected and it is not a determinism
regression: dropping the SVF's always-zero drive term lets `-ffast-math`
contract the band-pass update differently, which drifts the last bits by at
most 2.1e-6 absolute (~ -113 dBFS), measured over 1.9 M samples. On the desktop
(no `-ffast-math`) the new filter is bit-identical to the old one, and
`tests/test_svf_lp.cpp` pins that. Both repeat runs of this invocation agree
with each other, which is what the determinism check actually asserts.
